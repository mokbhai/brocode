# Phase 4: Write-Mode Guardrails and Output Review

## Goal

Make write-enabled automations safe and reviewable in both Local and Worktree modes.

## Scope

Add the guardrails and review metadata that turn scheduled code/file changes into auditable BroCode work:

- local dirty-checkout preflight
- advanced dirty-local opt-in
- worktree/branch preparation
- changed-file summary
- cancellation behavior
- retry behavior
- failure notes in result threads
- result-thread continuation

## Local Mode

Local mode is the default execution environment. It can make code and file changes when writes are enabled.

Before scheduled write-enabled local runs, the server must check the selected project checkout. If the checkout is dirty and the automation has not explicitly opted into dirty-local execution:

- do not dispatch the provider turn
- record a skipped or failed run with a precise reason
- append a note to the result thread if it exists, or create the result thread and append the note if practical

Manual `Run now` should use the same policy unless the UI provides an explicit one-time override.

Dirty-local opt-in should be an advanced automation setting, not the default.

## Worktree Mode

Worktree mode should create or reuse isolated branch/worktree state using existing BroCode thread/worktree helpers where possible. The implementation should avoid creating a separate Git orchestration stack for automations.

Worktree runs must persist:

- worktree path
- branch name
- base ref when available
- changed-file summary after run
- failure during preparation

Submission, merge, or PR creation remains human-controlled.

## Changed-File Summary

After terminal run states, write-enabled runs should record a changed-file summary. The summary should be lightweight:

- path
- status
- additions/deletions when cheaply available
- whether generated by local or worktree mode

Do not store large diffs in the automation run record. Link to existing diff/checkpoint surfaces where available.

## Cancellation

Cancel active automation run should:

- mark the run as cancelling/cancelled where possible
- dispatch existing thread/session stop or interrupt commands
- preserve enough state for the user to inspect partial output
- not delete the result thread

## Failure Notes

Failures should be visible in:

- run history
- automation list status
- result thread message/activity when practical

Failure notes should include enough context for recovery:

- what failed
- whether the provider ran
- whether files may have changed
- recommended next action

## Acceptance Criteria

- Scheduled write-enabled local runs skip by default when the checkout is dirty.
- Dirty-local opt-in allows local execution on dirty checkouts.
- Worktree mode prepares isolated execution state and persists branch/worktree metadata.
- Changed-file summaries are recorded after write-enabled terminal runs.
- Active runs can be cancelled from the UI.
- Failed and skipped runs produce user-visible recovery information.
- Result threads remain the central place to inspect automation output and continue work.

## Verification

- Git preflight tests for clean, dirty, untracked, and missing-repo states.
- Local policy tests for scheduled and manual triggers.
- Worktree preparation tests using existing Git/worktree helpers.
- Changed-file summary tests.
- Cancellation tests against fake provider/orchestration flows.
- Failure-note rendering tests.
